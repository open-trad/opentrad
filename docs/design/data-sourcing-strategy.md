# 选品数据源策略（三级可插拔）

**状态**：已采纳（发起人 2026-07-09 拍板"按三级数据源来"）
**调研日期**：2026-07-09 · 目标站点：1688 / 淘宝 / 拼多多 / Amazon / AliExpress

## 决策

选品/竞品数据层做成**三级可插拔 provider**，挂在 `packages/connectors` 后面。同一"商品搜索/详情"接口后面挂多个实现，按用户手里有什么资质/key、场景要多实时来路由。不追求一条路径打天下，也**不做纯协议级抓取**。

## 三级

### L0 — 浏览器适配器（bb-browser，已实现）
- **定位**：默认兜底 + 实时 + 登录态场景
- **用在**：需要登录态才能看的数据（1688 已登录价、拼多多个性化价）、实时交互式选品、官方/第三方 API 覆盖不到的兜底
- **为什么保留**：天然绕开签名与浏览器指纹两层（真页面 JS 生成签名、真浏览器指纹）——这是它相对协议级抓取的核心优势
- **成本**：0 现金；代价是依赖链长（Chrome + daemon + 登录态）、单机吞吐有限
- **现状**：11 站点已接（含发起人本地 1688/淘宝/拼多多/Amazon 私有适配器），插件页图形化启用

### L1 — 第三方数据 API（未实现，按需接）
- **定位**：批量后台选品主力补充
- **候选**：万邦 onebound（~0.04–0.1 元/次或包月数千）、Apify（$20/月+用量；1688 的 MOQ/供应商字段实测最全）、RapidAPI（Amazon 侧最成熟）
- **前置**：用户自带 provider key，产品不托管数据、不二次分发（规避主体风险，UI 需标"数据来源合规由用户负责"）；做 provider 抽象 + 字段映射统一到内部商品模型
- **合规**：灰色。靠"开源自用 + 用户自带 key + 不二次分发"稀释风险

### L2 — 官方/联盟 API（未实现，发起人自带资质时点亮）
- **定位**：高质量合规升级档，非默认
- **关键事实**：这几个平台的"任意商品搜索"几乎都不通过"商品 API"开放，而是通过**联盟/客 API**（淘宝客/多多进宝/阿里妈妈/AliExpress Affiliate/Amazon Creators API），需要推广者身份，返回"可推广商品池"而非全量；真正全量/交易 API 需商家授权，只适合抓自己店铺
- **子情况**：Amazon 卖家→SP-API；有联盟/流量来源→Amazon Creators API（**PA-API 已 2026-05 关停**）+ 淘宝客 + 多多进宝 + AliExpress Affiliate；1688 企业采购商→跨境铺货 API
- **成本**：调用多免费，真实成本是每站资质获取 + 授权维护 + 接入工时

## 明确不做

**纯协议级/无头 HTTP 抓取**（尤其淘系/拼多多的 x-sign/mtop 逆向）。它不降低依赖链，只是把"浏览器依赖"换成"签名逆向 + JA4 指纹跟版 + 代理池 + 封号处理"，长期维护成本更高、更易大面积掉线。淘系 2025 已上天盾 3.0；curl-impersonate 能过 JA3/JA4 但过不了业务层签名与账号风控。若要探索，仅限 Amazon/1688 这类签名较轻的站做实验性 provider，且明确标注"尽力而为、随时可能失效"。

## 路由优先级（同一"商品搜索"请求）

```
需要登录态 / 强实时 / API 覆盖不到  → L0 浏览器适配器
用户填了官方/联盟 key（该站）        → L2 官方/联盟 API（质量+合规最佳）
批量后台、用户填了第三方 key         → L1 第三方数据 API
以上都没有                          → 回落 L0，或提示"配置数据源"
```
